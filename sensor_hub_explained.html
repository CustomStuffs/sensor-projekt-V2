Sensor Hub — Project Explained

A guide for developers new to embedded + full-stack projects
A note before you read: This project was not designed by a professional engineer. It was built by someone with an idea — and maybe some clients — who figured things out along the way. Some of the rules and design choices in here might not be textbook-perfect. That is completely fine. The number one priority of this codebase is that it is always easy to add new features or sensors, even for someone who did not write the original code. If something looks unconventional, that is probably why — simplicity and extensibility were chosen over technical purity.

What does this project do?

It is an outdoor plant and environment monitor. A small microcontroller sits outside, reads physical sensors (soil moisture, humidity, light, etc.), and automatically controls a water pump. You can see all the live data in a web browser and send manual commands from anywhere in the world.

[Physical world — soil, air, light] | [Pico W microcontroller] ──── WiFi ────► [Raspberry Pi server] ────► [Your browser] reads sensors stores all data shows charts controls relay serves the dashboard sends commands

The three pieces (plus hardware)

The project is split into four folders. Each one is independent — a rule in this project is that you never change one without updating the others when the shared API contract changes.

1 Firmware — the tiny computer outside

The Raspberry Pi Pico W is a chip the size of a thumb drive. It has no screen or keyboard — it just runs a Python program on repeat. The language is MicroPython, a lightweight version of Python 3 designed for microcontrollers.

The main loop

The entry point is firmware/src/main.py. Everything the device does fits into this loop, which repeats every 60 seconds (development) or 30 minutes (battery mode): 

while True:
    reading = read_all_sensors(...)          # ask every sensor for a number
    commands, new_cfg = run_upload_cycle(...) # WiFi on → send data → get orders
    ack_id = handle_relay(commands, ...)     # turn pump on or off
    power.sleep(interval_s, relay=relay)    # deep sleep to save battery

Sensors — one file each

Every sensor lives in its own file inside firmware/src/sensors/. Each file has a single read() function that returns either a number (float) or None if the sensor is not connected or fails. It must never crash the program — a broken sensor should not stop the whole device.

Extensibility rule: Adding a new sensor = one new file in sensors/ + one new entry in config.json. Nothing else changes.

The config file

firmware/src/config.json is the only file you hand-edit. It holds WiFi credentials, which sensors are enabled, calibration numbers, and relay rules: 

{
  "wifi": { "ssid": "MyNetwork", "password": "secret" },
  "poll_interval_s": 60,
  "sensors": {
    "humidity":    { "enabled": true },
    "soil":        { "enabled": true, "adc_pin": 26 },
    "water_level": { "enabled": true, "pin": 16 }
  },
  "relay_rules": [
    { "sensor": "soil_pct", "op": "<", "value": 30,
      "action": "relay_on", "duration_s": 300 }
  ]
}

The last rule reads: "If soil moisture drops below 30%, run the pump for 5 minutes." No code change needed — just edit this file and reboot the device.

Ring buffer — offline storage

If WiFi fails for several hours, readings are kept in a ring buffer stored in the Pico W's flash memory (up to 48 readings). The next time WiFi succeeds, all buffered readings are uploaded at once. Nothing is lost.

2 Server — the brain on the Raspberry Pi

The server is a web API — a Python program that listens on port 8080 and answers HTTP requests. Both the Pico W and your browser talk to it. It is built with FastAPI, which automatically generates documentation and validates all incoming data.

File layout

server/

│   ├── main.py          ← starts the server, registers all routes
│   ├── models.py        ← defines what valid JSON looks like (Pydantic)

│       ├── time_sync.py ← POST /api/time
│       ├── readings.py  ← POST /api/readings, GET /api/readings
│       ├── commands.py  ← relay command queue
│       └── schedule.py  ← watering schedule
├── storage/
│   ├── database.py      ← SQLite connection + query helpers
│   └── schema.sql       ← table definitions
└── config/
    └── settings.py      ← port number, file paths

The database

All data lives in a single SQLite file (sensor_hub.db). SQLite is a database that is just one file on disk — no separate server process to manage. Perfect for a Raspberry Pi.

TableWhat it stores
devicesOne row per Pico W, with when it was last seen
readingsEvery sensor reading, forever — never deleted
commandsRelay commands queued by the dashboard
device_configWatering schedule per device
Idempotency trick: The readings table has UNIQUE(device_id, ts). If the Pico W retries the same upload because WiFi dropped mid-request, the server silently ignores the duplicate. No double counting.

Security — no passwords needed

There is no login system. Instead, the Raspberry Pi uses Tailscale — a VPN that makes the server only reachable from your own devices, over an encrypted tunnel. One command to set up, nothing to leak.

3 Dashboard — what you see in the browser

The dashboard is plain HTML, CSS, and JavaScript — no React, no Vue, no build step. The FastAPI server serves it as static files. When you open the URL in a browser, you get the dashboard; it then makes fetch() calls back to the same server to load data.

4 Hardware — the circuit board

A custom PCB (circuit board) connects all the sensors and the relay to the Pico W's GPIO pins. Designed in KiCad (free, open-source PCB design tool). Sensors plug in via Wago lever connectors — no soldering required to swap a sensor.

Key GPIO pin assignments

GPIO pinWhat's connected
GP2 / GP3I2C bus — light sensor, pH chip
GP8DS18B20 digital temperature probe (1-Wire)
GP9DHT22 humidity + temperature sensor
GP10Relay — drives the water pump
GP11PIR motion sensor (also wake-up interrupt)
GP26Soil moisture sensor (analog voltage)

How one complete cycle works — end to end

1.  Pico W wakes from deep sleep
2.  Reads all enabled sensors → builds a dict: { ts, temp_c, soil_pct, ... }
3.  Turns WiFi on
4.  POST /api/time  →  server replies with current unix timestamp + any config changes
5.  POST /api/readings  →  server saves the reading to SQLite
6.  GET  /api/commands  →  server replies with any pending commands
            (e.g. "relay_on for 300 s" because you clicked the button in the browser)
7.  Pico executes the command → turns the pump on
8.  WiFi off, back to deep sleep
9.  Next wake-up: POST /api/commands/42/ack  →  tells server "done, command executed"

Meanwhile in the browser:
    setInterval(() => fetchReadings(), 30_000)  →  charts update every 30 seconds

The API contract — the shared language

An API contract is an agreement about what JSON data is sent and received. The Pico W, the server, and the dashboard all need to agree — like three people speaking the same language. Here are the two most important endpoints:

POST /api/time — time sync and config delivery

Request (Pico → server):
  { "device_id": "pico_001", "uptime_ms": 123456 }

Response (server → Pico):
  { "unix_ts": 1748000000,
    "config": { "interval_s": 1800, "relay_schedule": [] } }

POST /api/readings — sensor data upload

Request (Pico → server):
  { "device_id": "pico_001", "fw": "1.0.0", "vbus_mv": 4850,
    "readings": [{ "ts": 1748000000, "temp_c": 22.5,
                   "humidity_pct": 65.2, "soil_pct": 45,
                   "ph": null, "lux": null }] }

Response: { "ok": true, "stored": 1 }
All sensor fields can be null. If a sensor is not connected, it simply sends null and the dashboard skips it. The pipeline handles missing sensors gracefully at every layer.

Key design decisions and why

Not all of these decisions came from a textbook. Some were made because they felt right for a solo builder with a small budget and real clients. The goal was always: keep it simple enough that you can come back to it in 6 months and still understand it.

DecisionWhy
One file per sensor, one config entry to enable it The most important rule in the whole project. New sensors or features should never require digging through unrelated code.
MicroPython, not C Much faster to write and debug. The Pico W is fast enough for sensor polling.
SQLite, not PostgreSQL Runs on a Pi with zero setup. Append-only sensor data fits SQLite perfectly.
Vanilla JS, no framework The dashboard is simple enough that plain JS is less code, not more.
Tailscale instead of auth One command to install. No passwords to manage or leak.
Ring buffer on the Pico Offline resilience — readings survive multi-hour WiFi outages.
None on sensor error, never raise A broken sensor must not crash the whole device.
config.json as the only tuneable file Non-programmers can adjust intervals, thresholds, and schedules without touching code.

Where to start if you want to change something

The whole codebase was designed around one principle: a new developer (or the original creator six months later) should be able to add something without having to understand everything first. Here is how that works in practice:

Add a new sensor: Create firmware/src/sensors/my_sensor.py with a read() function returning float | None. Add one entry to config.json. Done — the rest of the pipeline picks it up automatically. You do not need to touch the server or the dashboard just to start collecting the data.
Add a new API endpoint: Create a route file in server/api/routes/, register it in server/api/main.py, then add the matching fetch() call in dashboard/src/api.js. Update all three CLAUDE.md files to reflect the new contract.
Change the relay rules: Edit relay_rules in firmware/src/config.json and reboot the Pico. No code change needed.
If something feels wrong or too complicated: trust that instinct. This project was not built all at once by a team of experts — it grew from an idea. If a rule or pattern does not make sense to you, it might genuinely not be the best approach. The goal is always a system that works and that real people can maintain, not one that looks impressive on paper.
